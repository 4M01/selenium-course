In large test suites, you need quick access to different page URLs. HashMap provides instant lookup by page name.

Your test suite needs to navigate between Login, Dashboard, Profile, Settings, and Checkout pages. Instead of hardcoding URLs everywhere, you create a HashMap navigation system where you can simply say "Go to Dashboard" and get the correct URL instantly!

// Real Selenium Code Example HashMap<String, String> pageUrls = new HashMap<>(); // Set up your page navigation map pageUrls.put("Login", "https://myapp.com/login"); pageUrls.put("Dashboard", "https://myapp.com/dashboard"); pageUrls.put("Profile", "https://myapp.com/user/profile"); pageUrls.put("Settings", "https://myapp.com/settings"); pageUrls.put("Checkout", "https://myapp.com/checkout"); // Navigation helper method public void navigateToPage(String pageName) { String url = pageUrls.get(pageName); if (url != null) { driver.get(url); System.out.println("Navigated to " + pageName + ": " + url); } else { System.out.println("Page not found: " + pageName); } } // Usage in tests navigateToPage("Login"); // Goes to login page navigateToPage("Dashboard"); // Goes to dashboard

HashSet for Unique Element Tracking

Real Selenium Scenario: Tracking Visited Pages & Unique Elements

Track unique pages visited, unique error messages encountered, or unique test IDs to avoid duplicate processing.
üîç Selenium Reality: Smart Duplicate Prevention

You're crawling a website and testing every unique link, but some links appear multiple times on different pages. HashSet ensures you test each unique URL only once, saving time and avoiding duplicate test runs!

// Real Selenium Code Example HashSet<String> visitedUrls = new HashSet<>(); HashSet<String> uniqueErrorMessages = new HashSet<>(); // Track visited pages to avoid re-testing public boolean shouldVisitPage(String url) { if (visitedUrls.contains(url)) { System.out.println("Page already tested: " + url); return false; } else { visitedUrls.add(url); System.out.println("Testing new page: " + url); return true; } } // Collect unique error messages during testing public void checkForErrors() { List<WebElement> errorElements = driver.findElements(By.className("error-message")); for (WebElement error : errorElements) { String errorText = error.getText(); if (uniqueErrorMessages.add(errorText)) { System.out.println("New error found: " + errorText); // Log new error for analysis } } }

‚Ä¢ Automatic duplicate prevention

Streams for Smart Element Filtering

Real Selenium Scenario: Advanced Element Processing

Filter, transform, and process web elements using Java Streams for more efficient and readable automation code.
‚ö° Selenium Reality: Smart Element Processing Pipeline

You have 50 buttons on a page, but you only want to click the visible, enabled ones that contain 'Submit'. Streams let you create a smart filtering pipeline: find all buttons ‚Üí keep only visible ones ‚Üí keep only enabled ones ‚Üí keep only those with 'Submit' text ‚Üí click them!

// Real Selenium Code Example - Smart Element Processing List<WebElement> allButtons = driver.findElements(By.tagName("button")); // Traditional approach (verbose and hard to read) List<WebElement> clickableSubmitButtons = new ArrayList<>(); for (WebElement button : allButtons) { if (button.isDisplayed() && button.isEnabled() && button.getText().toLowerCase().contains("submit")) { clickableSubmitButtons.add(button); } } // Streams approach (clean and readable) List<WebElement> clickableButtons = allButtons.stream() .filter(WebElement::isDisplayed) // Only visible elements .filter(WebElement::isEnabled) // Only enabled elements .filter(btn -> btn.getText().toLowerCase().contains("submit")) // Only submit buttons .collect(Collectors.toList()); // Collect results // Click all filtered buttons clickableButtons.forEach(WebElement::click); System.out.println("Clicked " + clickableButtons.size() + " submit buttons");
Why Streams for Element Processing?
‚Ä¢ Clean, readable filtering logic
‚Ä¢ Chain multiple conditions easily
‚Ä¢ Functional programming approach
‚Ä¢ Perfect for complex element selection criteria

// This is what your complete system looks like in real Selenium code: public class TestManager { // Fixed test categories (Array) private String[] testCategories = {"Smoke Tests", "Regression Tests", "Integration Tests", "UI Tests"}; // Dynamic test cases (ArrayList) private ArrayList<String> testCases = new ArrayList<>(); // Test results mapping (HashMap) private HashMap<String, String> testResults = new HashMap<>(); // Unique bugs found (HashSet) private HashSet<String> uniqueBugs = new HashSet<>(); // Add test case to suite public void addTestCase(String testCase) { testCases.add(testCase); } // Update test result public void updateTestResult(String testCase, String result) { testResults.put(testCase, result); } // Report unique bug public boolean reportBug(String bugId) { return uniqueBugs.add(bugId); // Returns false if already exists } // Generate comprehensive report using Streams public void generateReport() { System.out.println("=== TEST EXECUTION REPORT ==="); // Stream processing for analysis long passedTests = testResults.values().stream() .filter(result -> result.equals("PASS")) .count(); long failedTests = testResults.values().stream() .filter(result -> result.equals("FAIL")) .count(); System.out.println("Total Test Cases: " + testCases.size()); System.out.println("Passed: " + passedTests); System.out.println("Failed: " + failedTests); System.out.println("Unique Bugs Found: " + uniqueBugs.size()); } }
